# Resume Hub: System Architecture & Implementation Report

This document provides a comprehensive overview of the Resume Hub project, designed for a new AI or developer to quickly understand, maintain, and scale the system.

---

## 1. Executive Summary
The **Resume Hub** is an AI-driven full-stack platform for parsing, indexing, and searching student resumes. It transforms a collection of raw PDFs and JSONs into a searchable, visualized knowledge base using the Gemini API and FAISS vector indexing.

---

## 2. System Architecture

### High-Level Flow
1.  **Ingestion**: Resumes (PDFs) are uploaded via the web UI.
2.  **Parsing**: A 5-call Gemini pipeline extracts structured JSON data (Skills, Work Exp, Projects, Scores).
3.  **Indexing**: Extracted text is distilled, embedded via `models/gemini-embedding-001`, and stored in a `FAISS` vector database.
4.  **Retrieval**: Users perform natural language queries. The system performs semantic search + LLM-based re-ranking with reasoning.
5.  **Visualization**: A knowledge graph (Vis.js) maps connections between students, skills, and companies.

### Technology Stack
- **Backend**: Python (FastAPI)
- **Frontend**: React (Vite) + Vis.js + CSS (Glassmorphism)
- **AI/ML**: Google Gemini 2.0/1.5, FAISS (Vector DB), NumPy
- **Deployment**: Docker, Render.com (Blueprint support)

---

## 3. Core Components

### 3.1 Backend (`backend/main.py`)
The FastAPI server acts as the orchestrator. Key endpoints:
- `GET /api/students`: Returns all parsed profiles.
- `POST /api/search`: Performs semantic search + AI re-ranking.
- `POST /api/upload`: Handles multi-file PDF uploads and triggers the parsing/indexing pipeline.
- `GET /api/graph`: Generates node-edge data for the knowledge graph.

### 3.2 AI Search Engine (`resume_search.py` integration)
The search logic uses a hybrid approach:
- **Vector Search**: Finds the top N candidates by cosine similarity.
- **LLM Reasoning**: Passes the "Distilled" profiles of the top candidates to Gemini to re-rank them based on the specific intent of the query and provides an explanation for each.

### 3.3 KeyRotator Logic
To handle high-volume parsing without hitting 429 Rate Limits, the system uses a `KeyRotator` class. It cycles through a pool of Gemini API keys, automatically retrying on failure with the next available key.

## 4. Key Implementation Changes

### Deployment Fixes
- **Gitignore Optimization**: Initially, the `frontend/dist` and `resume_index.faiss` were ignored. These were force-added to the repo to allow Render (on a free tier without persistent disk) to serve the app and index immediately.
- **Render Blueprint**: Created `render.yaml` to enable one-click deployment with support for Python runtime and environment variables.

### Search Optimization
- **Deduplication**: Fixed an issue where students appeared twice (once from `output/` and once from `manual_text/`).
- **Default Result Count**: Reduced the default search result count from 15 to 5 for cleaner AI reasoning outputs.
- **UI Aesthetics**: Implemented a dark glassmorphism theme with consistent proportional node sizing in the knowledge graph.

## 5. Deployment & Maintenance

### Environment Variables
The system requires `GEMINI_API_KEYS` (comma-separated). This is the only critical configuration needed to make the AI features work.

### Local Execution
```bash
# Backend
cd backend
python -m uvicorn main:app --reload

# Frontend
cd frontend
npm run dev
```

### Docker
The `Dockerfile` is a multi-stage build:
1.  **Frontend Build**: Compiles React to static assets.
2.  **Python Runtime**: Serves the FastAPI app and the static frontend assets on port 8000.

## 6. Future Scalability
- **Persistent Storage**: For production, move from local FAISS/JSON files to a managed Vector DB (like Pinecone) and a NoSQL DB (MongoDB).
- **Authentication**: Current version is for internal/demo use. Integration with Firebase or Clerk is recommended for multi-tenant use.
